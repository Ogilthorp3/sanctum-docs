The Memory Vault is Sanctum's shared knowledge base, inspired by how human memory works вҖ” which is to say, it forgets things on purpose and considers this a feature. It uses a three-layer architecture вҖ” working memory, short-term, and long-term вҖ” with a daily consolidation process that acts like sleep: distilling raw observations into lasting knowledge.

At some point during development, we realized we'd built a system that remembers where you left your SSH keys, forgets trivial session data, and dreams at 4 AM. The horror movie writes itself.

<Aside type="note">
  This documentation is automatically kept in sync with the Memory Vault codebase.
</Aside>

## Memory Stack

Sanctum runs a four-layer memory stack. Each layer does one thing well. Together they give the haus a memory system more sophisticated than most startups' data architecture.

| Layer | Role | Backend | Cost |
|-------|------|---------|------|
| **Mem0** | Primary cross-agent semantic memory | [mem0.ai](https://mem0.ai) via Rube/Composio | **Free** |
| **Memory Vault** | Git-native versioned source of truth | [sanctum-memory](https://github.com/Ogilthorp3/sanctum-memory) repo | Free |
| **Neo4j / Graphiti** | Entity relationships & graph queries | Local, port 31416 | Free (self-hosted) |
| **Supermemory** | Legacy cross-LLM bridge (secondary) | supermemory.ai MCP | Connected but secondary |

<Aside type="tip">
  **Mem0 is the primary memory layer.** It's free, supports structured filtering (by user, agent, project, metadata), and is wired into Rube where all automations live. Project: `sanctum-memory`, user: `Ogilthorp3`. Every recipe that needs context about the haus queries Mem0 first.
</Aside>

When Mem0 and the vault disagree, the vault wins вҖ” because Git has receipts. When the vault and the graph disagree, someone has a long night ahead of them. This has happened twice.

## Note Format

Every note has standardized YAML frontmatter. Opinions may vary on whether your haus needs a metadata schema for its thoughts. Opinions are wrong.

```yaml
---
type: semantic | episodic | procedural | observation | session_summary
source_agent: claude-code | gemini-cli | openclaw | user | system
created: 2026-03-20T17:00:00Z
updated: 2026-03-20T17:00:00Z
last_accessed: 2026-03-20T17:00:00Z
access_count: 3
importance: 0.85
consolidation_status: raw | reviewed | consolidated | archived
ttl_days: null  # null = permanent, or integer days
superseded_by: null  # path to newer version
tags: [network, infrastructure]
related_entities: [mac-mini, ubuntu-vm]
---

# Note Title

Content with [[wikilinks]] to related notes...
```

## Importance Scoring

Every note has a computed importance score (0.0 to 1.0) that determines its TTL. The vault is, in essence, deciding what's worth remembering. A power you'd think we'd reserve for sentient beings, but here we are.

| Importance | TTL | Notes |
|-----------|-----|-------|
| > 0.8 | Permanent | Core knowledge, user-stated facts |
| 0.5 - 0.8 | 90 days | Agent-observed patterns |
| 0.3 - 0.5 | 30 days | Single observations |
| < 0.3 | 7 days | Ephemeral session data |

**Score formula:** `source_weight x recency x access_frequency x link_density`

Notes accessed 5+ times are protected from expiry regardless of score. If the system keeps coming back to a memory, there's probably a reason. Even if that reason is paranoia.

## Consolidation

The consolidation engine runs daily at 4:17 AM (LaunchAgent: `com.sanctum.memory-consolidate`). Like sleep for the brain, except it runs on schedule, never hits snooze, and files a report when it's done.

<Steps>
1. **Scan inbox** for notes older than 24 hours
2. **Deduplicate** against existing long-term knowledge
3. **Classify** and move to the appropriate long-term directory
4. **Recompute** importance scores for all active notes
5. **Expire** notes that have exceeded their TTL
6. **Push to Mem0** вҖ” consolidated knowledge synced to Mem0 for cross-agent access
7. **Clean archive** вҖ” delete archived notes older than 90 days
8. **Generate report** in `meta/consolidation-log.md`
</Steps>

Any agent can trigger consolidation manually via the `memory_consolidate` MCP tool.

## Hard Caps

To prevent bloat, each layer has a maximum note count. Without limits, a system that remembers everything eventually remembers nothing useful вҖ” a problem familiar to anyone who's ever hoarded browser tabs.

| Layer | Max Notes | Action at Cap |
|-------|-----------|---------------|
| Inbox | 300 | Emergency consolidation |
| Knowledge | 1000 | Merge related notes |
| Events | 500 | Summarize old episodes |
| Procedures | 200 | Merge similar runbooks |

## Anti-Bloat Rules

- If derivable from code, logs, or sensors вҖ” don't store it
- If the same fact is stated multiple ways вҖ” dedup into one
- If never accessed in 30 days and importance < 0.5 вҖ” archive
- Raw data goes to time-series DBs, not memory
- Every episodic memory needs context (who, what, when, why)
- Every semantic memory should be self-contained

<Aside type="caution">
  These rules exist because the first version of the vault grew to 2,000 notes in a week and became functionally useless. A memory system that remembers everything is just a hard drive with pretensions.
</Aside>
