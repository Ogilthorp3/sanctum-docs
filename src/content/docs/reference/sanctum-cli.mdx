`sanctum` is the operator surface for Family Pass v1.0. It lives at `/opt/sanctum/sanctum-cli/bin/sanctum` and is symlinked into your `$PATH` during install. Nine subcommands cover the entire surface â€” health, regression, observability, upgrades, removal, and key custody.

```bash
sanctum <subcommand> [args]
```

Every command short-circuits cleanly when `~/.sanctum/sanctum.yaml` is missing. The errors are deliberately blunt â€” Family Pass is for technical-friend haushelds, and we trust you with the FATAL line rather than wrapping it in apology.

<Aside type="note">
There is no global `--verbose`. The subcommands are tight enough that adding one would mostly produce stack traces. If you need more than `sanctum doctor` and `sanctum logs <service>` tell you, the source is in `/opt/sanctum/sanctum-cli/lib/` and it is short enough to read end-to-end in one coffee.
</Aside>

## sanctum doctor

The first command you run, the first command you run when something is wrong, and the first command we will ask you to run when you reach out for help. It checks the host prerequisites and the subsystems that have to be up for anything else to work.

```bash
sanctum doctor
```

### What it checks

| Check | Pass condition |
|-------|----------------|
| `macOS` | `>=14` (Sonoma or later) |
| `Apple Silicon` | `uname -m == arm64` |
| `RAM` | `>=16 GB` |
| `brew` | Present on `$PATH` |
| `limactl` | Present on `$PATH` |
| `socket_vmnet` | Installed at `/opt/socket_vmnet/bin/socket_vmnet` |
| `Lima '<instance>'` | Status `Running` (instance name from `services.lima.instance`, default `openclaw-staging`) |
| `Firewalla @<host>` | `ping -c 1` reachable (host from `firewalla.host`, default `192.168.1.1`) |

### Exit codes

| Code | Meaning |
|------|---------|
| `0` | All checks passed |
| `1` | One or more checks failed |
| `1` (FATAL message) | `~/.sanctum/sanctum.yaml` not found |

<Aside type="tip">
`sanctum doctor` is wired into the install flow as the post-flight verification. If you forget which command to run after an unexpected reboot, it is this one.
</Aside>

### Mocked mode

For CI runs and `bats` tests, set `SANCTUM_DOCTOR_MOCK=1`. The doctor will print synthetic pass lines for the host triplet (`macOS 14.5`, `arm64`, `16 GB`) and skip the live checks. This exists so tests on dev laptops with the wrong RAM count do not fail spuriously. Not for use in production.

## sanctum self-test

The regression suite. After install, after `sanctum update`, after a Mac reboot, or whenever you suspect something has drifted, this is the command that tells you whether the eleven canonical probes still pass.

```bash
sanctum self-test
```

### The eleven probes

| # | Probe | Verifies |
|---|-------|----------|
| 1 | `01-network` | Macâ†”VM bridge interface is up |
| 2 | `02-firewalla-bridge-bearer-auth` | Bridge accepts the bearer token |
| 3 | `03-firewalla-rule-roundtrip` | A rule can be created and deleted via the bridge |
| 4 | `04-lima-vm-up` | The named Lima instance is `Running` |
| 5 | `05-neo4j-responding` | OpenClaw's graph store accepts a query |
| 6 | `06-devices-yaml-parses` | `~/.sanctum/devices.yaml` is valid YAML and has the expected shape |
| 7 | `07-schedule-fires` | The schedule cron actually wakes up and writes a heartbeat |
| 8 | `08-pwa-serves` | The Screen Time PWA returns 200 on `/` |
| 9 | `09-sops-roundtrip` | `sops` can encrypt and decrypt against the haushold age key |
| 10 | `10-age-key-roundtrip` | `age` can encrypt and decrypt without `sops` in the path |
| 11 | `11-firewalla-baseline-survives-bridge-down` | Killing the bridge does not drop the Firewalla baseline rules |

### Pass threshold

The suite passes when **at least 95% of probes succeed** â€” i.e. ten of eleven. The threshold is intentional: a single transient probe failure (the Firewalla under load, the Lima VM mid-respawn) should not make the suite red. Two failures should.

```
Summary: 11/11 probes passed
Result: PASS (>=95% threshold)
```

If self-test fails, look at the line immediately above `Summary:` to find which probe came back red. Then run the corresponding `lib/*.sh` or look at `sanctum logs self-test` for the last full transcript.

<Aside type="caution">
The 95% threshold does not mask probe 11. If `11-firewalla-baseline-survives-bridge-down` fails, you are running with no fail-closed enforcement â€” that is a fix-now condition even if the other ten passed.
</Aside>

### Mocked mode

| Env var | Behavior |
|---------|----------|
| `SANCTUM_SELF_TEST_MOCK=all_pass` | All probes return success without executing |
| `SANCTUM_SELF_TEST_MOCK=one_fail` | Probe 5 (`neo4j-responding`) returns failure; others succeed |
| `SANCTUM_SELF_TEST_MOCK=1` | Randomized pass/fail (regression test the threshold itself) |

### When to run

- After install (the installer runs it automatically).
- After `sanctum update`.
- After any macOS major version upgrade.
- After a Lima reboot where you are not certain the VM came back clean.
- Once a week as a hygiene pass, even if nothing seems wrong.

## sanctum status

The thirty-second health glance. Six lines, color-blind safe `[ok]` / `[FAIL]` markers, no narrative. If you only run one command in the morning, run this one.

```bash
sanctum status
```

### What it checks

| Service | Probe |
|---------|-------|
| Lima VM | `limactl list openclaw-staging --format '{{.Status}}'` returns `Running` |
| Firewalla bridge | `curl localhost:1984/health` with bearer token returns 200 |
| Screen Time daemon | `launchctl list` shows `com.sanctum.screen-time` |
| Force Flow | TCP probe to `127.0.0.1:4077` |
| Backup plist | `launchctl list` shows `com.sanctum.backup` |
| PWA | `curl localhost:5151/` returns 200 |

### Mocked mode

`SANCTUM_STATUS_MOCK=1` returns synthetic `[ok]` for every line. Tests only.

`sanctum status` is faster than `sanctum doctor` and exists for the case where you just want to know whether your haus is healthy right now. `doctor` is what you reach for when `status` shows red. The two are deliberately not the same command.

## sanctum update

Pulls every Sanctum repo, restarts the LaunchAgents that need restarting, then re-runs `sanctum self-test`. Designed to be run by hand when there is a new release; not (yet) wired into a launchd timer.

```bash
sanctum update
sanctum update --dry-run
```

### What it does

1. `git pull --ff-only` in each of:
   - `/opt/sanctum/sanctum-runtime`
   - `/opt/sanctum/sanctum-screen-time`
   - `/opt/sanctum/sanctum-cli`
2. `launchctl kickstart -k` for `com.sanctum.screen-time` and `com.sanctum.firewalla`.
3. `sanctum self-test`.

### Flags

| Flag | Behavior |
|------|----------|
| `--dry-run` | Print every action without executing |

If a pull fails (uncommitted local changes, divergent history) the update halts at that repo. Sanctum does not stash your changes for you. The assumption is that if you have edits in `/opt/sanctum/`, you know what they are and you would rather we ask.

<Aside type="tip">
If `sanctum self-test` fails immediately after an update, the most common cause is a probe that expects a new env var you have not set yet. Check the release notes before reaching for `git revert`.
</Aside>

### Mocked mode

`SANCTUM_UPDATE_MOCK=1` skips the actual `git pull`, `launchctl kickstart`, and `self-test` invocations. Tests only.

## sanctum uninstall

Removes the LaunchAgents, stops the Lima VM, and clears Firewalla baseline rules. Preserves your config and state by default. Pass `--purge` if you actually want everything gone.

```bash
sanctum uninstall
sanctum uninstall --purge
```

### What it removes (always)

- All Sanctum LaunchAgents (`com.sanctum.vm-autostart`, `com.sanctum.force-flow`, `com.sanctum.backup`, `com.sanctum.fileproviderd-watchdog`, `com.sanctum.screen-time`, `com.sanctum.firewalla-bridge-watchdog`, `com.sanctum.firewalla`)
- The Lima `openclaw-staging` instance (stopped, not deleted)
- Firewalla rules tagged `sanctum-baseline` (best-effort, fail-quiet)

### What it preserves (without `--purge`)

- `~/.sanctum/` (sanctum.yaml, devices.yaml, schedule.yaml, age key bundle, backup archives)
- `~/.openclaw/` (agent configs, log history)
- Keychain entries (Firewalla bridge token, Anthropic key)

### What `--purge` adds

- `rm -rf ~/.sanctum`
- `rm -rf ~/.openclaw`

`--purge` does **not** delete Keychain entries. Use `security delete-generic-password -s firewalla-bridge-token` etc. if you want those gone too. The asymmetry is intentional â€” files inside `~/.sanctum` are easy to back up before purging; Keychain entries are not, and we would rather you delete them deliberately than accidentally.

<Aside type="caution">
`--purge` is irreversible. If you have not exported your age key (`sanctum keys backup`), any sops-encrypted secrets in your backups become unrecoverable the moment `~/.sanctum` goes away. There is no recovery email.
</Aside>

### Mocked mode

`SANCTUM_UNINSTALL_MOCK=1` skips the `launchctl bootout`, `limactl stop`, and Firewalla DELETE calls but still removes config files when `--purge` is set. Tests only.

## sanctum logs &lt;service&gt;

A typed shortcut around `tail -F` for the logs you actually look at. Resolves a short name to the right path under `~/.openclaw/logs/`, then execs `tail -F`. Ctrl-C to exit, same as you would expect.

```bash
sanctum logs <service>
```

### Service names

| Name | File |
|------|------|
| `backup` | `~/.openclaw/logs/sanctum-backup.log` |
| `vm-autostart` | `~/.openclaw/logs/vm-autostart.log` |
| `force-flow` | `~/.openclaw/logs/force-flow.log` |
| `screen-time` | `~/.openclaw/logs/screen-time.log` |
| `firewalla` | `~/.openclaw/logs/firewalla-bridge.log` |
| `self-test` | `~/.openclaw/logs/sanctum-self-test.log` |

Anything else is rejected with the list of valid names. We resisted the temptation to make this fuzzy-match â€” there are six services, and you should know which one you want.

<Aside type="tip">
If a log file does not exist, the command exits with a FATAL line rather than silently waiting for the file to appear. A missing log means the LaunchAgent never wrote one, which is itself the diagnostic.
</Aside>

## sanctum devices

List, add, or remove devices in `~/.sanctum/devices.yaml`. This is the file Screen Time uses to map MAC addresses to human-readable names and per-device schedule categories.

```bash
sanctum devices
sanctum devices list
sanctum devices add <MAC> <NAME> [CATEGORY]
sanctum devices remove <MAC>
```

### Subcommands

| Subcommand | Behavior |
|------------|----------|
| `list` (or no arg) | Print all devices in `devices.yaml`, formatted `<mac>  <name>  category=<category>` |
| `add MAC NAME [CATEGORY]` | Append a device. `CATEGORY` defaults to `other`. |
| `remove MAC` | Remove the device whose `mac` matches. No prompt. |

```bash
sanctum devices add aa:bb:cc:dd:ee:ff "Albert's iPad" school
sanctum devices list
sanctum devices remove aa:bb:cc:dd:ee:ff
```

The file is rewritten in place via PyYAML. Comments and key order are not preserved across writes â€” if you have hand-edited `devices.yaml` with annotations, mutate it from your editor instead and Sanctum will re-read it on the next probe cycle.

<Aside type="note">
The override env var is `SANCTUM_DEVICES_YAML`. Used by `bats` to point at a fixture instead of your live devices file. Production paths should leave this alone.
</Aside>

## sanctum schedule

Show or edit `~/.sanctum/schedule.yaml`, the file that defines wind-down windows, curfew, homework gates, and screen-time credits.

```bash
sanctum schedule
sanctum schedule show
sanctum schedule edit
```

### Subcommands

| Subcommand | Behavior |
|------------|----------|
| `show` (or no arg) | Cat the schedule file to stdout |
| `edit` | Open the schedule file in `$EDITOR` (default: `nano`) |

`schedule edit` does not validate the file on save. If you write invalid YAML, the Screen Time probe (`07-schedule-fires`) will catch it on the next tick. The fast feedback loop is "edit, save, run `sanctum self-test`" â€” about ten seconds.

<Aside type="tip">
The full schema for `schedule.yaml` lives in the [Screen Time docs](/screen-time/). This command just opens the file; it does not generate one. If `~/.sanctum/schedule.yaml` does not exist, the installer's Screen Time module writes the default at first boot.
</Aside>

The override env var is `SANCTUM_SCHEDULE_YAML`.

## sanctum keys backup &lt;path&gt;

Export an encrypted bundle of your haushold age key. This is the single most important command in the CLI. Without this bundle, a hardware failure makes every sops-encrypted secret in your Sanctum unrecoverable.

```bash
sanctum keys backup ~/Desktop/sanctum-keys.age
```

### What it does

1. Reads your age key from `~/.config/sops/age/keys.txt` (override: `SANCTUM_AGE_KEYS`).
2. Calls `age -p` to passphrase-encrypt the key. You will be prompted for the passphrase interactively.
3. Writes the encrypted bundle to the path you specified.

### What it does not do

- It does not auto-generate a passphrase. You choose one. The CLI will not save it for you.
- It does not copy the bundle anywhere. It is on disk, exactly where you told it to go.
- It does not call any cloud service. The encryption is local, with `age`, against a passphrase you typed once and must remember.

```bash
# Encrypt
sanctum keys backup ~/Desktop/sanctum-keys.age

# Decrypt (later, on a different machine if you need to restore)
age -d -o keys.txt ~/Desktop/sanctum-keys.age
```

<Aside type="danger">
The recovery model is: you have the bundle, you have the passphrase, you can restore. If either is gone, the encrypted half of your Sanctum is gone. There is no recovery email. There is no support line that can decrypt this for you. Print the passphrase on paper if you have to. We are deadly serious about this.
</Aside>

## Options

Two global options. Anything else gets routed to the subcommand.

| Option | Behavior |
|--------|----------|
| `--version` | Print the CLI version and exit `0` |
| `-h`, `--help` | Print the usage banner and exit `2` |

```bash
sanctum --version    # â†’ sanctum 1.0.0
sanctum --help       # â†’ usage banner
```
