Sanctum manages a set of macOS LaunchAgents (user-level) and LaunchDaemons (root-level) that form the boot chain for the haus intelligence platform. Each plist is rendered from templates at `~/.sanctum/templates/launchagents/` by the `generate-plists.sh` script, which pulls values from `instance.yaml` and tokens from the macOS Keychain.

Every LaunchAgent uses the `com.sanctum.*` reverse-DNS namespace. Every process shows a descriptive name in Activity Monitor. If you open Activity Monitor and see seventeen copies of `python` and `bash` with no indication which is which, you're running an older version. Upgrade. Your sanity is worth it.

You might wonder why these aren't written in a friendlier format. The answer is that Apple chose XML for process management configuration in 2005 and has been politely pretending that was fine ever since. The template system exists so you never have to touch raw plist XML. You're welcome.

<Aside type="tip">
Services with `KeepAlive: true` are automatically restarted by launchd if they exit. Services without KeepAlive run once at load and are not restarted. KeepAlive is the difference between "runs at boot" and "runs forever, whether it likes it or not."
</Aside>

## Process Naming

macOS Activity Monitor shows the process name from `argv[0]` — the first entry in `ProgramArguments`. When that entry is `/opt/homebrew/bin/node` or `/usr/bin/ssh`, you get a sea of identical process names and zero useful information about what's actually running.

Sanctum solves this with symlinks in `~/.sanctum/bin/`. Each service gets a named symlink that points to its real binary:

```bash
~/.sanctum/bin/sanctum-gateway     → /opt/homebrew/bin/node
~/.sanctum/bin/sanctum-ha-tunnel   → /usr/bin/ssh
~/.sanctum/bin/sanctum-memory-vault → ~/.sanctum/memory-vault/.venv/bin/python
```

Because the plist's `ProgramArguments[0]` is the symlink, launchd launches the real binary but Activity Monitor displays the symlink's filename. Open Activity Monitor, and you see `sanctum-gateway`, `sanctum-ha-tunnel`, `sanctum-memory-vault` — not `node`, `ssh`, `python`.

```bash
# Create or update all symlinks
~/.sanctum/scripts/create-bin-symlinks.sh

# Verify
ls -la ~/.sanctum/bin/
```

<Aside type="caution">
The Rust binaries (`sanctum-proxy`, `sanctum-watchdog`, `sanctum-idle`) are already compiled with descriptive names. They don't need symlinks. Only interpreted-language services (Node, Python, Bash, SSH) get the symlink treatment.
</Aside>

## Boot Chain Overview

LaunchAgents are loaded at user login. The ordering below reflects the logical dependency chain — launchd does not guarantee ordering, but `RunAtLoad: true` ensures all agents start promptly after login.

In practice, everything starts within seconds of each other and sorts itself out. It's less of a chain and more of a stampede in roughly the right direction.

## LaunchAgents

### Core Infrastructure

These agents stand up the VM, the gateway, and the firewall bridge. Without them, the rest of the stack is a collection of orphaned processes with nowhere to send their feelings.

#### com.sanctum.utm-autostart

| Property | Value |
|----------|-------|
| **Label** | `com.sanctum.utm-autostart` |
| **Activity Monitor** | `sanctum-utm-autostart` |
| **Purpose** | Launch UTM, start the VM, and set the bridge100 IP to the configured `mac_bridge_ip` |
| **Required Service** | `vm` |
| **KeepAlive** | No |
| **RunAtLoad** | Yes |

Runs a startup script that launches UTM.app, waits for the VM to boot, and configures the bridge interface IP via `sudo ifconfig`. Requires the `vmnet-bridge` sudoers entry at `/etc/sudoers.d/vmnet-bridge`.

The first domino. Everything else assumes the VM is running and the bridge exists. If this one fails, enjoy your very expensive space heater.

#### com.sanctum.gateway

| Property | Value |
|----------|-------|
| **Label** | `com.sanctum.gateway` |
| **Activity Monitor** | `sanctum-gateway` |
| **Purpose** | OpenClaw/DenchClaw agent gateway on the Mac side |
| **Required Service** | `gateway` |
| **KeepAlive** | Yes |
| **RunAtLoad** | Yes |
| **Port** | Configured via `services.gateway.port` |

The Mac-side agent gateway. The symlink `sanctum-gateway` points to `/opt/homebrew/bin/node` (stable Homebrew symlink, not a versioned Cellar path) because pointing a LaunchAgent at a Cellar path is a time bomb with a `brew upgrade` fuse.

<Aside type="caution">
Always restart via `openclaw gateway stop` and `openclaw gateway start`. Never use raw `launchctl` commands for this service — the CLI handles state cleanup. Ignoring this advice results in orphaned port locks and a gateway that thinks it's running when it isn't, which is worse than a gateway that's honestly down.
</Aside>

## LaunchDaemons

Everything above runs as your user. Everything below runs as root. There is exactly one daemon in this section, and it exists because of a number: 80. The lowest-numbered privilege escalation in the history of home automation.

### com.manoir.dench-proxy

| Property | Value |
|----------|-------|
| **Label** | `com.manoir.dench-proxy` |
| **Purpose** | Reverse proxy from port 80 to port 19001 for the Holocron chat interface |
| **Required Service** | `gateway` |
| **KeepAlive** | Yes |
| **RunAtLoad** | Yes |
| **Runs as** | `root` |

This is a **LaunchDaemon** (not a LaunchAgent) because binding to port 80 requires root privileges. It enables `http://holocron/` access from the LAN without specifying a port.

Plist location: `/Library/LaunchDaemons/com.manoir.dench-proxy.plist`

The entire reason this runs as root is so family members can type `holocron` into a browser instead of `holocron:19001`. Usability has a cost. That cost is `sudo`.

## Plist Generation

All plists are rendered from Mustache-style templates using values from `instance.yaml` and tokens from the macOS Keychain:

```bash
# Preview what would be generated (dry run)
~/.sanctum/generate-plists.sh --dry-run

# Generate and install all plists for enabled services
~/.sanctum/generate-plists.sh
```

The generator reads each `com.sanctum.*.plist` template from `~/.sanctum/templates/launchagents/`, checks if the corresponding service is enabled in `instance.yaml`, expands `{{PLACEHOLDER}}` tokens with config values, pulls secrets from the macOS Keychain, and writes the rendered plist to `~/Library/LaunchAgents/`.

## Managing Agents

Load or unload agents using `launchctl`:

```bash
# Load an agent
launchctl bootstrap gui/$(id -u) ~/Library/LaunchAgents/com.sanctum.proxy-watchdog.plist

# Unload an agent
launchctl bootout gui/$(id -u)/com.sanctum.proxy-watchdog

# Check if an agent is running
launchctl print gui/$(id -u)/com.sanctum.proxy-watchdog

# List all Sanctum agents
launchctl list | grep com.sanctum
```

<Aside type="caution">
The gateway (`com.sanctum.gateway`) should always be managed via `openclaw gateway start/stop`, not via raw `launchctl` commands. The CLI handles cleanup of state files and port locks. This is the third time these docs have told you this. There will not be a fourth.
</Aside>

## Migration from Legacy Labels

If you're upgrading from before the `com.sanctum.*` unification, run the one-shot migration script:

```bash
~/.sanctum/scripts/migrate-process-names.sh
```

This script unloads old agents (`ai.openclaw.*`, `com.manoir.*`, `com.user.*`, `com.yoda.*`), removes their plist files from `~/Library/LaunchAgents/`, creates the `~/.sanctum/bin/` symlinks, regenerates all plists from the new templates, and loads the new agents. Use `--dry-run` to preview changes first.

<Aside type="note">
The migration script is idempotent. Running it twice won't break anything — it'll just skip everything that's already done and report `[ok]` for each step. If only all migrations were this polite.
</Aside>
