## What blocks the first friendly beta tester

Ordered by impact (highest first):

### 1. sanctum-cli v0.8 productization

| Item | Status |
|---|---|
| `sanctum onboard --recipe family --yes` super-command | live shipped in v0.7.1 |
| 16-pattern secret scanner refusal | live shipped |
| GitHub Tier 0 (free private repo per host) | live shipped |
| iCloud Photos auto-exclude | live shipped |
| Brew formula (`brew install ogilthorp3/sanctum/sanctum-cli`) | pending targeted May, not yet shipped |
| Sigstore signing on binaries | pending targeted May |
| Public release on Ogilthorp3/sanctum-cli | pending same |
| Smoke-tested "fresh Mac to working R2 backup in 5 min" recipe | live verified Apr |

**What this enables**: a beta tester runs `brew install sanctum-cli && sanctum onboard --recipe family --yes` and gets a working stack. Without v0.8, install is operator-guided.

### 2. Cathedral long-context eval harness

**Status as of 2026-05-21**: Tasks 1-10 shipped on `feat/long-context-eval-harness`. Stdlib-only Python; the package lives at `services/sanctum-mlx/tools/long-context-bench/` with `niah.py` (NIAH probe), `ruler_subset.py` (5 mini-RULER tasks), `runner.py` (HTTP runner + scoring + trend + gate), `test_bench.py` (mocked-cathedral test suite, **38 tests passing**), the entrypoint shim, and the weekly LaunchAgent plus post-deploy gate wrapper.

Task 11 (full operating runbook) still to write. A live trend run against the cathedral was kicked off this afternoon to confirm end-to-end function; first row landing in `trend.jsonl` will mark the harness fully validated.

### 3. R2D2 has never fired in production

600+ classifier-only cycles, 0 false positives, 0 fires. The fire path is unit-tested but has never been validated against a real surfaced detection in the production environment. Before flipping out of classifier-only, the operator should:

- Wait for R2D2 to surface a real detection (most likely candidate: another stale-binary find after the next reboot)
- Manually validate that the proposed fix is correct
- Remove the `~/.sanctum/state/r2d2-classifier-only` marker for that one detection
- Watch it fire under operator supervision
- Re-create the marker

Until that's happened at least once, beta testers shouldn't see R2D2 in fire mode.

### 4. MBP-side automation key

Today's SSH automation-key fix is Mini-only. The MBP needs the same `~/.ssh/sanctum_automation_ed25519` (private key) plus the `Host manoir-automation` block in its `~/.ssh/config`. Three sanctum scripts on the MBP-side workspace would otherwise still hit 1Password unlock prompts.

Runbook: `runtime/docs/doctrine/runbooks/mbp-automation-key-setup.md`. ~3 minutes of operator-side work.

### 5. SanctumBridge.app code-signed bundle wrapper

The bridge runs on brew node (currently pinned). When the pin is eventually released to take a security patch, TCC will re-prompt for iMessage, Calendar, Contacts. A code-signed bundle around the brew node binary would key TCC to the bundle ID instead of the executable path — grants survive upgrades.

Requires the Apple developer cert workflow already used for other Sanctum binaries. ~half-day of work.

### 6. Family Pass v1.0 (fortnight plan)

`docs(plans): Family Pass v1.0 fortnight implementation plan` committed but not implemented. Multi-user / family-member scoping isn't real yet — sanctum is built around the operator's single account. For a beta tester who shares the haus with non-tech family, this matters.

Scope: per the committed plan. ~2 weeks of operator-driven implementation.

### 7. Screen-time Layer 2 (reconciler + reaper)

Layer 1 (honest API + bounded state) shipped on manoir. Layer 2 (reconciler that walks state vs reality, reaper that drops orphaned records) is the resilience tier. Separate repo: [`Ogilthorp3/sanctum-screen-time`](https://github.com/Ogilthorp3/sanctum-screen-time).

### 8. Beta-grade install walkthrough (documented + recorded)

End-to-end: fresh Mac → installed sanctum → first useful agent interaction → first autoheal cycle. Written walkthrough exists in fragments (sanctum-cli README, individual narrative pages). Consolidated walkthrough doesn't.

## What "first beta" actually looks like

Realistic profile: a single technical friend with a Mac, willing to follow a written setup guide, who runs Sanctum for one week and reports issues. Not a production deployment, not a multi-user family unit, not a turnkey product.

Minimum gates for first-beta:
- Items 1, 3, 4, 8 above must be closed.
- Items 2 (eval harness) is *strongly preferred* — without it, regressions land silently.
- Items 5, 6, 7 can land *during* the beta cycle, with the tester aware they're in flight.

Timing estimate from 2026-05-20:
- 2-3 weeks if items 1 + 8 ship at the targeted May/June pace.
- 4-6 weeks if item 2 (eval harness) is held as a hard prerequisite.

<Aside type="note">
The gap between "I run it" and "they run it" lives in writing, scripting, and trust gates — not in the infrastructure. Power-cycle, autoheal, log rotation, the safety stack — those are all done. What remains is the productization wrap and the validation harness. Both are crisp pieces of work with no architectural unknowns. This checklist gets updated as items land.
