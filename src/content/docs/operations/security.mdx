## Defense in Depth

Sanctum uses seven layers of security, because if action movies have taught us anything, it's that every single layer will eventually be breached by someone with enough determination. The game isn't making the wall perfect — it's making the attacker tired.

1. **Network isolation** — VM has no internet access (host-only networking)
2. **Firewall** — pf rules on LAN interface block unauthorized port access
3. **SSH hardening** — Key-only auth, no root login, AllowUsers restriction
4. **Encrypted secrets** — SOPS+age on VM, macOS Keychain on Mac (FileVault disabled — see below)
5. **Automatic rotation** — 8 secrets rotated monthly
6. **Service binding** — Services bound to specific interfaces (not 0.0.0.0)
7. **PII anonymization** — Personal data scrubbed from all external LLM requests

<Aside type="tip">
The VM literally cannot reach the internet. It sits in a digital room with no doors to the outside, talking to the Mac through a slot in the wall. This is not a metaphor. This is the actual network topology.
</Aside>

## pf Firewall

Rules in `/etc/pf.anchors/sanctum` block external access to internal ports on the LAN interface (en1):

Blocked ports include: gateway (1977), dashboard (1111), XTTS (8008), MLX (1337), Firewalla bridge (1984), and others.

The philosophy: if a service doesn't need to talk to the LAN, it doesn't get to. Every open port is a decision, not a default.

## Secret Rotation

Monthly via `com.sanctum.rotate-secrets` LaunchAgent (1st of month, 3:30am).

Rotates 8 secrets:
1. Home Assistant main + Windu tokens
2. Firewalla bridge token
3. Gateway token
4. Network control token
5. Backup encryption key
6. OpenRouter API key
7. Deepgram API key

Plus verifies the Cloudflare tunnel token.

```bash
# Manual rotation
bash ~/Backups/rotate-secrets.sh

# Dry run
bash ~/Backups/rotate-secrets.sh --dry-run
```

<Aside type="note">
The secrets rotate while you sleep. By the time you wake up, every token from last month is dead. This is the kind of thing that sounds paranoid until you read about the breach that happened because someone's API key was six months old and had been quietly exfiltrated in February.
</Aside>

## SSH Hardening

Both Mac and VM have hardened SSH configs:
- Key-only authentication (no passwords)
- No root login
- AllowUsers restriction
- Post-quantum key exchange (on VM)

Yes, post-quantum key exchange. For a VM that can't access the internet. Because when the quantum computers arrive, they'll find this particular door already bolted from the inside.

## VM Isolation

The Ubuntu VM runs on host-only networking (bridge100, 10.10.10.0/24). It can only reach the Mac at 10.10.10.1 — no direct internet access. All external communication goes through the Mac.

This means a compromised VM can't phone haus, can't exfiltrate data, can't join a botnet. The worst it can do is send a strongly worded message to the Mac, which is already watching.

## Secrets Storage

| Location | Purpose |
|----------|---------|
| macOS Keychain | Runtime token access |
| 1Password | Backup copies of all secrets |
| VM SOPS (age) | Encrypted secrets for VM services |

Secrets are **never** stored in config files, git repos, or environment variables on disk.

<Aside type="danger">
If you find a secret in a git repo, something has gone wrong. Rotate it immediately, then figure out how it got there, then have a quiet word with whoever committed it. Possibly yourself. Especially yourself.
</Aside>

## FileVault — Deliberately Disabled

FileVault is off on the Mac Mini. This was a council decision, not an oversight.

The Jedi Council convened on 2026-03-29 to evaluate whether full-disk encryption made sense for a stationary home server. The verdict was unanimous: **availability beats encryption for a device that never leaves the house.**

### Why It Is Off

FileVault protects against one threat: someone physically stealing the Mac Mini or removing its SSD. It creates a different problem: every power outage, every kernel panic, every Hydro-Quebec winter adventure turns the Mac Mini into a $600 paperweight until someone walks over and types a password. There is no remote unlock. There is no VPN trick. The FileVault pre-boot screen runs before the network stack exists. The Mac sits there, encrypted and useless, while the entire home automation infrastructure — security cameras, alarm system, climate control, voice agents — goes dark.

The council reasoning:

- **Windu (Security):** A deaf security agent is worse than an unencrypted disk. When the Mac is down, Signal alerts stop, security automations stop, Alarmo goes blind. The attack surface of a powered-off security system is larger than the attack surface of an unencrypted SSD in a locked house behind Firewalla.
- **Qui-Gon (Infrastructure):** The Mac Mini is the backbone of 30+ services across two nodes. Every minute of unattended downtime cascades. FileVault turned a recoverable event (power restored) into a manual intervention (drive to the house, type a password).
- **Yoda (Grand Master):** The Force flows through availability, not through encryption of a stationary box.

### Compensating Controls

FileVault is one layer of seven. Removing it shifts weight to the others:

1. **Physical security** — The Mac Mini is in a locked house, behind Firewalla, on a private LAN segment.
2. **Secret rotation** — All 8 secrets rotate monthly. If the device is ever physically compromised, rotation limits the blast radius.
3. **1Password backup** — All secrets have copies in 1Password, not just on the Mac disk.
4. **UPS recommended** — A basic UPS handles the 90% case (short power blips). FileVault removal handles the 10% case (extended outage, unattended recovery).
5. **Bootstrap LaunchDaemons** — Two LaunchDaemons (`com.sanctum.vmnet` + `com.sanctum.bootstrap`) start all services at boot without GUI login. The VM runs via `socket_vmnet` + bare QEMU. No auto-login needed, Touch ID preserved.

<Aside type="note">
If you are a security auditor reading this and feeling your eye twitch: the Mac Mini sits behind a Firewalla Purple, on a VLAN with no port forwarding, in a house with physical locks, in a city where the most common threat model is a squirrel chewing through a power line. We made the call. The squirrel won.
</Aside>

### Re-enabling FileVault

If the threat model changes (the Mac Mini moves to a co-lo, Bert becomes a person of interest, etc.):

```bash
sudo fdesetup enable
```

The disk will encrypt in the background over several hours. Remember that this re-introduces the remote-reboot problem — keep `fdesetup authrestart` in your muscle memory.

### Remote Reboot Protocol

With FileVault off, remote reboots just work:

```bash
# Standard reboot (SSH)
sudo reboot

# Graceful reboot with service drain
sudo shutdown -r +1 "Sanctum maintenance reboot"
```

The Mac boots, LaunchDaemons fire, all services start (including the VM via socket_vmnet), and the council reconvenes. No GUI login required. No drive to the house. No calling your neighbor. Touch ID stays.

## PII Anonymization

All requests routed to external LLM providers (OpenRouter) pass through a Presidio-based anonymization layer before leaving the network. This ensures personal data never reaches third-party inference APIs.

Your AI agents know your name. The cloud doesn't need to.

### How It Works

The Sanctum Proxy (port 4040) handles all LLM request routing. When a request targets an OpenRouter model — either directly or via fallback routing — the proxy:

1. Extracts text from user and assistant messages (system prompts are left untouched)
2. Sends the text to a local Presidio analyzer container to detect PII entities
3. Sends detected entities to a local Presidio anonymizer container to replace them with placeholders
4. Forwards the scrubbed request to the external provider
5. De-anonymizes the response before returning it to the caller

### What Gets Scrubbed

| Entity | Example | Replacement |
|--------|---------|-------------|
| Person names | `John Smith` | `<PERSON>` |
| Email addresses | `john@example.com` | `<EMAIL_ADDRESS>` |
| Phone numbers | `514-555-1234` | `<PHONE_NUMBER>` |
| Credit cards | `4111-1111-1111-1111` | `<CREDIT_CARD>` |
| SSN / bank numbers | `123-45-6789` | `<US_SSN>` |

Only entities above 0.7 confidence score are anonymized. IP addresses, locations, and code identifiers are intentionally excluded to avoid breaking technical context.

### What Is Not Affected

- **Anthropic (Claude)** — direct API, no PII scrubbing (trusted provider, privacy policy reviewed)
- **Local models** (LM Studio, Council-27B MLX) — never leave the network
- **Gemini** — routed via Google AI Studio API (separate trust decision)
- **System prompts** — contain instructions, not personal data

### Architecture

```
Client → Sanctum Proxy (port 4040) → Provider
              |                           |
              +-- Presidio Analyzer (Docker, port 5002)
              +-- Presidio Anonymizer (Docker, port 5001)
              |                           |
              +-- PII scrubbed before exit ◄──────────+
              +-- PII restored on response
```

### Presidio Containers

The analyzer and anonymizer run as local Docker containers, bound to localhost only:

```bash
# Check status
docker ps --filter name=presidio

# Restart if needed
docker restart presidio-analyzer presidio-anonymizer
```

<Aside type="caution">
If the Presidio containers are down, PII scrubbing silently stops working and requests go through unscrubbed. The watchdog checks for this, but it's worth knowing: the absence of a safety net looks exactly like the presence of one until you fall.
</Aside>
