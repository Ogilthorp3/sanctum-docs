This page is the current-state ledger. It is intentionally less romantic than the architecture pages. The architecture pages explain how Sanctum is supposed to work. This page records what was actually verified on the machine.

## Current Verified State

### Workspace-owned slice

- `sanctum/catalog.yaml` is the checked-in canonical source for the workspace service manifests
- `sanctum/render_services.py --check` reproduces the checked-in manifests deterministically
- `tests/test-sanctum-audit.sh` covers rendering, sidecar behavior, graph compatibility, and docs wiring
- `navigator-sidecar.js` exposes aggregate project status and scrubs obvious secret material from its payload

### Runtime slice

- `~/.sanctum/instance.yaml` validates cleanly
- `~/.sanctum/services/` now contains 32 rendered manifests: 27 instance-backed services plus 5 auxiliary runtime services derived from launchd wiring and the checked-in `sanctum/runtime_catalog.yaml`
- Runtime wrappers under `~/.sanctum/bin` are present
- LaunchAgent drift has been remediated to a clean baseline according to the custom auditor
- The runtime graph now carries explicit verified edges such as `xtts -> voice-agent` instead of treating every instance service as an isolated root

### Documentation slice

- Stable architecture doctrine lives primarily in [The Living Force](/architecture/living-force/)
- Dated changes and migration events live in [Operational History](/operations/operational-history/)
- Audit-style corrections live in [Implementation Audit](/operations/implementation-audit/) and [Runtime Drift Audit](/operations/runtime-drift-audit/)

## What This Means Practically

Sanctum is in better shape than it was when the audit started:

- one canonical checked-in service catalog for the workspace slice
- deterministic manifest generation
- a cleaner sidecar contract
- end-to-end audit coverage on the parts we touched
- runtime LaunchAgent drift repaired
- documentation boundaries made explicit

<Aside type="tip">
The system is now easier to reason about because fewer things are implicit. That is the whole game in infrastructure. Performance matters. Reliability matters. But clarity is what keeps you from debugging ghosts.
</Aside>

## What Is Still Not True

A few claims remain too generous if stated without qualifiers:

- Sanctum is not yet a single-repo system
- Sanctum is not yet a fully portable install experience
- The runtime layer still depends on local state outside the checked-in workspace

Those are not failures. They are the boundaries of the current design.
