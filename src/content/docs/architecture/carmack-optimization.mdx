Somewhere between the third hardware-induced kernel panic and the fourth "Broken Pipe" error at 2 AM, it became clear that the MLX training pipeline was not merely inefficient. It was actively hostile to the hardware. We were asking a Mac Mini to train a Council model while feeding it data shapes chaotic enough to make Metal itself file a grievance.

Following a full audit of the training path, inspired by John Carmack's preference for unified memory, hard measurements, and contempt for hand-wavy performance folklore, we rebuilt the pipeline around stable shapes and observable bottlenecks. The result is a training loop that behaves like software instead of a superstition.

## Phase 1: Sequence Bucketing (Compiler Protection)

**The Issue:** Training with `batch_size: 1` on unpadded datasets caused MLX to compile a new Metal shader for every unique sequence length. This JIT thrashing exhausted the `MTLCompilerService`, leading to "Broken Pipe" errors mid-training.

**The Fix:**
- `build_dataset.py` was refactored to calculate exact token lengths using the Qwen 3.5 tokenizer.
- All examples are now padded to fixed buckets: **512, 1024, or 2048** tokens.
- **Impact:** The compiler now sees only three graph shapes. Crashes disappeared, and CPU overhead dropped from "why is the laptop screaming" to something the operating system could discuss calmly.

## Phase 2: Kernel Fusion & Graph Defragmentation

**The Issue:** Deep LoRA layers and high gradient accumulation steps created an explosion of tiny, dynamic memory allocations, hitting the `[metal::malloc]` resource limit (499,000).

**The Fix:**
- Initially stabilized by reducing `num_layers` and `rank`.
- Once the graph shapes were locked (via Phase 1), we successfully pushed the configuration back to **32 layers** and a **2048 context window** without triggering OOM or allocation limit crashes.
- The computation engine now processes a highly stable, consolidated computational graph.

<Aside type="note">
The important detail here is not that we temporarily reduced ambition. It is that we reduced entropy first, then turned ambition back up. Performance tuning that begins with chaos is just faster chaos.
</Aside>

## Phase 3: Python GIL Bypass

**The Issue:** The `generate_synthetic.py` script relied on single-threaded Python string manipulation, bottlenecking dataset generation as the council expanded to 9 agents.

**The Fix:**
- Refactored `generate_synthetic.py` to use Python's `multiprocessing.Pool()`.
- **Impact:** Removed the Global Interpreter Lock (GIL) bottleneck, allowing the dataset generation to execute concurrently across all available CPU cores.

## Phase 4: Hardware Profiling Pipeline

**The Issue:** Configuration tweaks were based on trial and error rather than actual hardware metrics.

**The Fix:**
- Created `scripts/profile_training.sh`.
- Wraps the MLX training command with Apple Instruments: `xcrun xctrace record --template 'Metal System Trace'`.
- Allows real-time tracking of ALU occupancy, shader compilation time, and unified memory bandwidth utilization.

This matters because "it feels faster" is not a metric. It is how people end up benchmarking vibes.

## E2E Validation

A dedicated end-to-end test suite, `tests/test_e2e_pipeline.py`, enforces the rules the optimization depends on:

1. **Data Generation:** validates parallel execution
2. **Sequence Bucketing:** validates padding and dropping logic
3. **Live Inference:** sends HTTP payloads to the local inference server to prove the deployed adapter is actively streaming tokens

The inference check runs through `sanctum-server` on ports `1337` and `8901`, which proxy to `mlx_lm.server`. That matters because a training optimization is only real if the resulting adapter still serves tokens in the actual runtime path, not just in a notebook where everyone is on their best behavior.

## Relation To Sanctum-MLX

The Carmack Optimization targets the **training** pipeline. The **serving** side is evolving in parallel through `sanctum-mlx`, a Rust replacement for the Python `mlx_lm.server` backend.

Once `sanctum-mlx` gains LoRA adapter loading support, the hot path from HTTP request to Metal GPU to SSE response becomes zero-Python. That is appealing both technically and emotionally.
