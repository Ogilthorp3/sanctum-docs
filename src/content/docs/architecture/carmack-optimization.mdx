# The Carmack Optimization

**Date:** 2026-04-02
**Subject:** `mlx-finetune` Pipeline Architecture
**Status:** Implemented and Verified

Following an analysis of the training architecture—inspired by John Carmack's philosophy on unified memory and hardware profiling—several critical optimizations were implemented to prevent hardware crashes and maximize training efficiency for the **Mothma 9-agent Council** (Qwopus 27B).

## Phase 1: Sequence Bucketing (Compiler Protection)

**The Issue:** Training with `batch_size: 1` on unpadded datasets caused MLX to compile a new Metal shader for every unique sequence length. This JIT thrashing exhausted the `MTLCompilerService`, leading to "Broken Pipe" errors mid-training.

**The Fix:**
- `build_dataset.py` was refactored to calculate exact token lengths using the Qwen 3.5 tokenizer.
- All examples are now padded to fixed buckets: **512, 1024, or 2048** tokens.
- **Impact:** The compiler only sees 3 distinct graph shapes, eliminating the crashes and drastically reducing CPU overhead.

## Phase 2: Kernel Fusion & Graph Deframgmentation

**The Issue:** Deep LoRA layers and high gradient accumulation steps created an explosion of tiny, dynamic memory allocations, hitting the `[metal::malloc]` resource limit (499,000).

**The Fix:**
- Initially stabilized by reducing `num_layers` and `rank`.
- Once the graph shapes were locked (via Phase 1), we successfully pushed the configuration back to **32 layers** and a **2048 context window** without triggering OOM or allocation limit crashes.
- The computation engine now processes a highly stable, consolidated computational graph.

## Phase 3: Python GIL Bypass

**The Issue:** The `generate_synthetic.py` script relied on single-threaded Python string manipulation, bottlenecking dataset generation as the council expanded to 9 agents.

**The Fix:**
- Refactored `generate_synthetic.py` to use Python's `multiprocessing.Pool()`.
- **Impact:** Removed the Global Interpreter Lock (GIL) bottleneck, allowing the dataset generation to execute concurrently across all available CPU cores.

## Phase 4: Hardware Profiling Pipeline

**The Issue:** Configuration tweaks were based on trial and error rather than actual hardware metrics.

**The Fix:**
- Created `scripts/profile_training.sh`.
- Wraps the MLX training command with Apple Instruments: `xcrun xctrace record --template 'Metal System Trace'`.
- Allows real-time tracking of ALU occupancy, shader compilation time, and unified memory bandwidth utilization.

## E2E Validation

A dedicated End-to-End Test Suite (`tests/test_e2e_pipeline.py`) was built to enforce these standards:
1.  **Data Generation:** Validates parallel execution.
2.  **Sequence Bucketing:** Validates padding and dropping logic.
3.  **Live Inference:** Sends HTTP payloads to the local `mlx_lm.server` to prove the deployed adapter is actively streaming tokens.
